# Phase 8.2: Deep Integration - Research

**Researched:** 2026-03-28
**Domain:** Container lifecycle bridging to real tmux sessions
**Confidence:** HIGH

## Summary

Phase 8.2 is the critical missing bridge between the v2 container/supervisor system (900+ unit tests, fully built) and actual tmux session management. Currently, the supervision tree creates AgentContainers and transitions their FSMs to "running", but no tmux session is ever created by the container lifecycle. The health tree faithfully reports phantom states. Discord commands (`/dispatch`, `/kill`, `/relaunch`) route through the supervision tree but only manipulate FSM state, never launching or killing real Claude Code sessions.

The fix is conceptually simple but touches multiple integration points: AgentContainer.start() must call TmuxManager.create_session() + send the Claude Code launch command. Container.stop() must kill the tmux session. Health reports must check actual tmux liveness (not just FSM state). The supervisor monitor must detect tmux session death and transition containers to ERRORED. Additionally, `/status` must be removed and `/health` confirmed as the replacement.

**Primary recommendation:** Bridge AgentContainer lifecycle methods to TmuxManager operations, add a periodic tmux liveness check in the supervisor monitor loop, and remove `/status`.

## Architecture Patterns

### The Gap: Container Lifecycle vs Real Tmux

Current flow (broken):
```
/dispatch → company_root._find_container() → reports FSM state "running"
           (but no tmux session was ever created by the container)

Supervisor._start_child() → create_container() → container.start()
           (transitions FSM creating→running, opens MemoryStore, but nothing else)
```

Required flow:
```
Supervisor._start_child() → create_container() → container.start()
           → TmuxManager.create_session("vco-{project}")
           → TmuxManager.create_pane(session, agent_id)
           → TmuxManager.send_command(pane, claude_launch_cmd)
           → Store pane_id on container for liveness checks

container.stop() → TmuxManager.kill_session/pane → FSM transition
container.health_report() → check tmux pane is actually alive
```

### Pattern 1: TmuxBridge on AgentContainer

**What:** Give AgentContainer (or a mixin/wrapper) access to TmuxManager and pane references so lifecycle methods bridge to real tmux operations.

**When to use:** Every container that manages a real Claude Code session.

**Approach:** Add optional `tmux_manager` and `pane_id` attributes to AgentContainer. Override `start()` to create tmux session/pane and launch Claude Code. Override `stop()` to kill the tmux pane. Override `health_report()` to check tmux liveness.

```python
# In AgentContainer or a subclass:
class AgentContainer:
    def __init__(self, ..., tmux_manager: TmuxManager | None = None):
        self._tmux: TmuxManager | None = tmux_manager
        self._pane: libtmux.Pane | None = None
        self._pane_id: str | None = None

    async def start(self) -> None:
        self._lifecycle.start()
        await self.memory.open()
        if self._tmux is not None:
            await self._launch_tmux_session()

    async def _launch_tmux_session(self) -> None:
        # Create pane in project session, send claude command
        ...

    async def stop(self) -> None:
        if self._tmux is not None and self._pane_id:
            self._kill_tmux_pane()
        self._lifecycle.stop()
        await self.memory.close()

    def is_tmux_alive(self) -> bool:
        if self._tmux is None or self._pane_id is None:
            return False
        pane = self._tmux.get_pane_by_id(self._pane_id)
        return pane is not None and self._tmux.is_alive(pane)
```

**Key insight:** TmuxManager must be injected into the container creation path. The factory (`create_container`) and supervisor (`_start_child`) need to pass it through.

### Pattern 2: Supervisor Liveness Monitoring

**What:** The supervisor's `_monitor_child` loop currently waits on `asyncio.Event` for FSM state changes. It also needs to periodically check tmux liveness to detect when a Claude Code session crashes without the FSM knowing.

**Approach:** Add a periodic `asyncio.wait` with timeout to the monitor loop. On timeout, check tmux liveness. If the tmux pane is dead but the container FSM says "running", transition to ERRORED.

```python
async def _monitor_child(self, child_id: str) -> None:
    event = self._child_events[child_id]
    while True:
        try:
            await asyncio.wait_for(event.wait(), timeout=30.0)
            event.clear()
        except asyncio.TimeoutError:
            # Periodic liveness check
            container = self._children.get(child_id)
            if container and container.state == "running":
                if hasattr(container, 'is_tmux_alive') and not container.is_tmux_alive():
                    await container.error()
                    # error() triggers state change callback → event set
            continue
        # ... existing failure handling ...
```

### Anti-Patterns to Avoid

- **Dual state sources:** Do not let the container FSM and tmux liveness diverge permanently. The liveness check must reconcile them.
- **Blocking tmux calls in async context:** TmuxManager operations are synchronous (libtmux). Must use `asyncio.to_thread()` for any blocking operations in the async bot/supervisor context.
- **Modifying existing v2 module internals:** Keep changes to the bridge layer. Do not rewrite the FSM, supervisor strategies, or health aggregation.

## Common Pitfalls

### Pitfall 1: Blocking libtmux calls in async context
**What goes wrong:** libtmux is synchronous. Calling TmuxManager methods directly from async supervisor/container code blocks the event loop.
**Why it happens:** TmuxManager was designed for CLI use (sync). Bot/supervisor code is async.
**How to avoid:** Wrap all TmuxManager calls in `asyncio.to_thread()`. Example: `await asyncio.to_thread(tmux.create_session, name)`.
**Warning signs:** Bot becomes unresponsive during dispatch/kill operations.

### Pitfall 2: Race between tmux death detection and FSM state
**What goes wrong:** Tmux session dies, liveness check fires, transitions container to ERRORED, supervisor restart creates new container, but the old liveness check is still running for the old container reference.
**Why it happens:** Monitor task references stale container after restart.
**How to avoid:** Cancel monitor task before restarting child (already done in `_start_child`). Verify the container reference is current before acting on liveness check.

### Pitfall 3: Phantom containers from /new-project
**What goes wrong:** `/new-project` creates containers via add_project() but the containers start() without tmux bridging, creating phantoms.
**Why it happens:** add_project() → supervisor.start() → _start_child() → container.start() has no tmux awareness.
**How to avoid:** TmuxManager must be injected into the supervisor/container creation path so _start_child creates real sessions.

### Pitfall 4: /status removal breaks test expectations
**What goes wrong:** Removing /status breaks tests that reference it.
**Why it happens:** test_commands_cog.py likely has tests for /status.
**How to avoid:** Remove /status tests when removing the command. Verify with grep.

### Pitfall 5: Environment variables for Claude Code launch
**What goes wrong:** Container launch fails because DISCORD_BOT_TOKEN or other env vars are not available.
**Why it happens:** The bot process has these env vars, but the tmux session may not inherit them.
**How to avoid:** Explicitly pass env vars in the launch command (matching dispatch_cmd.py pattern).

### Pitfall 6: Workspace trust prompt blocks Claude Code
**What goes wrong:** Claude Code shows workspace trust prompt, agent hangs waiting for user input.
**Why it happens:** First launch in a new clone directory.
**How to avoid:** Send empty enter key after 3s delay (matching dispatch_cmd.py pattern). Use `--dangerously-skip-permissions` flag.

```python
async def start(self) -> None:
    self._lifecycle.start()
    await self.memory.open()
    if self._tmux is not None:
        await self._launch_tmux_session()

async def _launch_tmux_session(self) -> None:
    session = await asyncio.to_thread(
        self._tmux.get_or_create_session,
        self._project_session_name,
    )
    pane = await asyncio.to_thread(
        self._tmux.create_pane,
        session,
        window_name=self.context.agent_id,
    )
    self._pane_id = pane.pane_id
    # Build launch command from context
    launch_cmd = self._build_launch_command()
    await asyncio.to_thread(self._tmux.send_command, pane, launch_cmd)
    # Auto-accept workspace trust
    await asyncio.sleep(3)
    await asyncio.to_thread(pane.send_keys, "", enter=True)
```

## Open Questions

1. **Should TmuxManager be a singleton or per-supervisor instance?**
   - What we know: TmuxManager wraps libtmux.Server(). Multiple instances share the same tmux server.
   - What's unclear: Thread safety of multiple TmuxManager instances.
   - Recommendation: Single TmuxManager instance created in VcoBot, passed through the tree. Simpler, no thread safety issues.

2. **Should FulltimeAgent (PM) and CompanyAgent (Strategist) also get tmux sessions?**
   - What we know: PM and Strategist are event-driven, not GSD pipeline agents. They react to events via Discord.
   - What's unclear: Whether they need Claude Code sessions.
   - Recommendation: Only GsdAgent and ContinuousAgent need tmux sessions. PM/Strategist operate through the Anthropic SDK, not tmux. Add a guard in start(): only launch tmux for types that need it, or use an attribute on the container type.

3. **How does `/dispatch` interact with already-running containers?**
   - What we know: Currently /dispatch just reports FSM state.
   - Recommendation: If container is already running with a live tmux session, report state. If stopped/errored, trigger restart through supervisor (which will create new tmux session).

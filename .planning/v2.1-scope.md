# v2.1 — Behavioral Integration

**Goal:** Make the v2.0 container infrastructure operational. Agents do real work, PM leads development through conversational gates, health tree shows the full picture.

**Principle:** No new infrastructure. Wire what exists. Every gap below has the structural code already built — this milestone connects it.

---

## Gaps to Close

### CRITICAL — Agents must do work

**GAP-01: Work Initiation**
- `start_workflow()` must send a GSD command (`/gsd:discuss-phase N`) to the agent's tmux pane, not just post a Discord message
- Container needs readiness detection (wait for Claude prompt before sending command)
- `_build_launch_command()` may need an `--initial-prompt` or post-launch command delivery

**GAP-02: Auto Work Distribution**
- When an agent completes a phase/milestone item, PM assigns the next one
- BacklogQueue.claim_next() exists but nothing calls it
- Idle GsdAgent should poll or be pushed its next assignment from ProjectStateManager
- "GSD begins on injected milestone automatically" — the auto-assignment loop

**GAP-03: PM Event Routing (Full)**
- Health state changes (agent RUNNING→ERRORED) must be routed to PM's event queue, not just Discord #alerts
- GSD state transitions (DISCUSS→PLAN, PLAN→EXECUTE) must be routed to PM
- Briefings from ContinuousAgents must be routed to PM
- Escalations (agent BLOCKED) must be routed to PM
- Currently only `task_completed` is routed

### CRITICAL — PM-Led Development Gates

**GAP-15: Conversational PM Review Gates**
The core development flow is PM-led through Discord conversation:

1. Agent finishes a job (plan, task, phase)
2. Agent posts to its channel: `[agent-id] @PM, I have finished Mile 1 Phase 2 planning, need your review` (attaches key files)
3. PM reviews attached files against its memory/context and responds:
   - **Approve:** `[PM] Looks good, go on!` → Agent continues
   - **Modify:** `[PM] I see issues, fix X to follow Y structure...` → Agent fixes → back to review
   - **Clarify:** `[PM] I don't understand how X relates to Y, give me context` → Multi-turn conversation → PM proposes alternative → Agent continues
4. Gate is conversational — not a button press. PM and agent talk until PM is satisfied.
5. Only then does the agent advance to the next GSD stage.

**Implementation needs:**
- Agent knows to stop and ask PM after each GSD stage transition
- PM receives the message, reads attached files, forms opinion, responds
- Agent reads PM's response, decides if approved/needs-changes/needs-discussion
- Multi-turn conversation loop until gate passes
- All of this happens in the agent's Discord channel
- Message throttling: 1 message per 30 seconds per agent (Discord rate limit friendliness)

**GAP-16: PM Memory and Context for Reviews**
- PM needs to read and understand plans/code that agents produce
- PM needs persistent memory of project decisions, architecture choices, past reviews
- PM needs to be opinionated — not rubber-stamp, actually review quality

### MAJOR — System architecture

**GAP-04: Strategist as Container**
- `CompanyAgent._handle_event()` is `pass` — all Strategist logic lives in StrategistCog
- Strategist should operate through the container event queue, not as a standalone Cog
- StrategistCog becomes a thin Discord adapter; logic moves to CompanyAgent

**GAP-05: Communication Port Wiring**
- DiscordCommunicationPort exists but `comm_port=None` in all container creation paths
- Wire it during container creation in both `client.py` on_ready and `commands.py` /new-project
- Agents use comm_port for structured messaging (not raw channel.send)

**GAP-06: BLOCKED as FSM State**
- Currently a Python bool, invisible to health tree
- Make it a real FSM state: any running state → BLOCKED → resume previous state
- Health tree shows BLOCKED with reason

**GAP-07: PM Outbound Actions**
- PM can only receive events, has zero outbound triggers
- Needs: trigger integration, inject milestone, request agent recruitment/removal via supervisor, escalate to Strategist
- These are method calls on PM that route through the supervision tree

**GAP-08: Supervision Hierarchy**
- Strategist should be direct child of CompanyRoot, peer to ProjectSupervisors
- Currently all agents go under ProjectSupervisor regardless of type
- CompanyRoot needs to manage company-scope agents separately

### MINOR — Completeness

**GAP-09: Health Tree Full Rendering**
- Show CompanyRoot at top with its state

- Show ProjectSupervisor level

**GAP-10: STOPPING State**

- Allows graceful shutdown visibility in health tree

**GAP-11: ContinuousAgent Delegation**
- DelegationTracker exists and works
- ContinuousAgent needs a `request_task()` method that calls supervisor.handle_delegation_request()

**GAP-12: ContinuousAgent Persistent Memory**
- Only cycle_count persisted currently
- Add: seen_items, pending_actions, briefing_log, config to memory_store

**GAP-13: Assignment Recovery on Restart**
- GsdAgent restores FSM state from checkpoint
- Must also restore work context: current phase, current task, what files to work on
- Agent re-reads assignment from ProjectStateManager on restart

**GAP-14: Stuck-Agent Detection**
- PM monitors how long each agent has been in the same GSD state
- If stuck > threshold, PM intervenes via Discord message to agent channel
- Threshold configurable per agent type

## Success Criteria

After v2.1, this must work:

1. `/new-project` → agents start working autonomously on Phase 1
2. Agent completes plan → posts to channel with files → PM reviews → approves or requests changes → agent continues
3. Agent completes phase → PM assigns next phase from backlog
4. Agent crashes → supervisor restarts → agent resumes from checkpoint with full work context
5. `/health` shows: CompanyRoot → Strategist → ProjectSupervisor → PM + agents with inner states
6. Agent gets stuck → PM detects → sends message to unblock
7. All inter-agent communication flows through CommunicationPort and MessageQueue (throttled)

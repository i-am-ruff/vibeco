2. Create `docker/settings.json` -- a minimal default Claude Code settings file:
   ```json
   {
     "permissions": {
       "allow": [
         "Bash(*)",
         "Read(*)",
         "Write(*)",
         "Edit(*)",
         "Glob(*)",
         "Grep(*)",
         "WebFetch(*)"
       ],
       "deny": []
     }
   }
   ```
   This gives Claude Code full tool access inside agent containers. Per D-09, this lives inside the container's writable layer at /root/.claude/settings.json.

3. Create `docker/Dockerfile` with this exact content:
   ```dockerfile
   # vCompany Claude Code agent image
   # Base: node:22-slim (D-01) with Python 3.12, git, tmux, uv
   # One universal image for all agent types (D-03)
   FROM node:22-slim

   # System dependencies: Python 3.12, git, tmux, curl (for uv installer)
   # Single RUN to minimize layers; clean apt cache
   RUN apt-get update && apt-get install -y --no-install-recommends \
       python3 python3-venv python3-pip \
       git tmux curl ca-certificates \
       && rm -rf /var/lib/apt/lists/*

   # Install uv (Python package manager)
   RUN curl -LsSf https://astral.sh/uv/install.sh | sh
   ENV PATH="/root/.local/bin:$PATH"

   # Install Claude Code globally (D-02)
   RUN npm install -g @anthropic-ai/claude-code

   # Apply tweakcc patches at build time (D-02) -- deterministic, no runtime surprises
   RUN npx tweakcc --apply

   # Default Claude settings baked into image (D-09)
   # Container writable layer preserves modifications across stop/start
   RUN mkdir -p /root/.claude
   COPY settings.json /root/.claude/settings.json

   # Working directory mounted at runtime via -v host_clone:/workspace
   WORKDIR /workspace

   # Keep container alive for exec-based interaction
   # tmux server started by DockerTransport.setup() via docker exec
   CMD ["sleep", "infinity"]
   ```

   Layer ordering rationale: system deps first (rarely change), then npm installs (change on Claude Code updates), then tweakcc (change on patch updates), then settings (change on config updates). This maximizes Docker build cache hits.

4. Note: Do NOT run `docker build` in this task. The image build is a verification step, not a build-time requirement. The executor should verify the Dockerfile is syntactically valid but the actual build can be slow and is not required for the plan to proceed. If Docker is available, a quick `docker build --check` or similar can validate syntax.

# v3.0 — CLI-First Architecture Rewrite

## Problem

All core logic lives inside the Discord bot process (`on_ready`, cogs). The CLI can't access containers, supervision tree, or state. This means:
- Strategist can't hire/task/dismiss agents via `vco` commands
- No programmatic access without Discord
- Bot crash = total system failure (no graceful degradation)
- Testing requires a running Discord bot
- Logic is duplicated between CLI and bot (clone, init vs /new-project)

## Principle

**CLI owns all logic. Bot is a Discord skin. Same as how agents use `vco report`.**

## Target Architecture

```
vco CLI (owns all logic)
├── vco up          → starts runtime daemon
├── vco init        → create project structure
├── vco clone       → clone repos, deploy artifacts
├── vco hire        → create task agent with full container lifecycle
├── vco give-task   → queue task for agent (idle-gated delivery)
├── vco dismiss     → stop and clean up agent
├── vco status      → show health tree
├── vco report      → agent posts status to Discord
└── vco new-project → init + clone + hire all agents (replaces bot /new-project)

Runtime daemon (long-lived, started by vco up)
├── CompanyRoot + supervision tree
├── Container lifecycle + task queues
├── Idle signal watchers
├── Health monitoring loop
└── Local API (Unix socket)
    └── All vco commands hit this API when runtime is up

Discord Bot (thin UI layer, started alongside runtime)
├── /new-project    → calls vco new-project
├── /health         → calls vco status, formats as embed
├── /standup        → calls vco status per agent, creates threads
├── on_message #strategist → relay to Strategist conversation
├── on_message #agent-*    → calls vco give-task (owner messages)
├── on_message #task-*     → calls vco give-task (Strategist messages)
└── PM review flow  → calls vco give-task after gate resolution
```

## Key Changes

### Phase 1: Extract runtime
- CompanyRoot, supervisors, containers, health monitoring → standalone daemon
- Local Unix socket API (JSON protocol)
- `vco up` starts both runtime daemon and bot
- Bot connects to runtime via socket

### Phase 2: CLI as API client
- All `vco` commands become thin clients to the runtime API
- `vco hire researcher market-scout` → socket call → runtime creates container
- `vco give-task market-scout "Research X"` → socket call → runtime queues task
- `vco status` → socket call → runtime returns health tree

### Phase 3: Bot as thin relay
- Slash commands call `vco` CLI (subprocess) or hit runtime API directly
- `on_message` handlers call `vco give-task` for relay
- No container references in bot memory
- No supervision tree in bot — just API calls

### Phase 4: State persistence
- Container state, pane IDs, task queues persist to disk
- Runtime restart recovers all containers from saved state
- Bot restart has zero impact on running agents

### Phase 5: Strategist autonomy
- Strategist calls `vco hire`, `vco give-task`, `vco dismiss` via Bash
- No special wiring, no action tag parsing, no callbacks
- Same commands work for owner in terminal, Strategist in claude -p, bot in Discord

## What This Unblocks

- Strategist hires agents via Bash (no Discord round-trip, no parsing)
- Owner manages agents from terminal without Discord
- Bot crash doesn't lose agent state
- Integration tests run without Discord or bot
- Multiple UIs can connect to same runtime
- Clean separation of concerns

## Known Issues & Gaps (from this session)

### Strategist Cannot Execute Actions

**Problem:** The Strategist runs `claude -p` with `--allowedTools "Bash Read Write"`. It has Bash access. But `vco hire`, `vco give-task`, `vco dismiss` don't exist as CLI commands yet. The bot's CompanyRoot.hire() exists in Python but is only callable from within the bot process.

**Current workaround (broken):** We added `[CMD:hire ...]` action tag parsing in StrategistCog._execute_actions(). This parses text from the Strategist's response and executes it. Problems:
1. The Strategist outputs `[CMD:hire ...]` as literal text in Discord instead of executing it — it's a claude -p response, not a tool call
2. The parsing code exists in the working tree but isn't deployed (bot hasn't restarted)
3. Even if deployed, it's fragile — depends on the LLM formatting the tags exactly right
4. **This is the wrong approach entirely.** The Strategist should call `vco hire` via Bash, same as agents call `vco report`.

**Fix (requires v3.0):** `vco hire` CLI command that hits the runtime API. Strategist calls it via Bash.

**Interim fix (pre-v3.0):** Could add `vco hire` as a CLI command that writes a command file, bot watches and executes. Ugly but functional.

### Strategist Model Was Wrong

**Problem:** StrategistConversation was running `claude -p` without `--model` flag. Defaulted to Sonnet instead of Opus. Result: Strategist was noticeably dumber, didn't push back on ideas, skipped conversation phases, went straight to research.

**Fix (done):** Added `--model opus` to all `claude -p` command builders. Default is `model="opus"` in constructor. Session bumped to v10.

### Strategist Persona Not Conversation-First

**Problem:** Despite detailed persona instructions ("probe, challenge, push back, ask hard questions", "Do NOT rush", "Never generate project files from a single vague message"), the Strategist immediately started researching when given a vague product idea. Didn't ask a single probing question.

**Likely causes:**
1. Wrong model (Sonnet — fixed, see above)
2. Persona loaded as first USER message, not system prompt — may have less weight
3. The `_STYLE_REMINDER` re-injected every 10 messages is about formatting, not behavior
4. `--allowedTools "Bash Read Write"` means the model is tool-biased — it has Bash so it wants to use it

**Potential fixes:**
- Switch persona from user message to `--system-prompt` flag (but loses prompt caching)
- Add a behavioral reminder alongside the style reminder: "REMEMBER: Challenge the idea first. Do NOT research or build until you've asked hard questions and the scope is clear."
- Consider `--append-system-prompt` for the persona instead of sending as first message
- Opus should be significantly better at following persona instructions than Sonnet

### Task Agent Readiness Detection Untested E2E

**Problem:** The signal-based idle detection (Stop/SessionStart hooks writing to sentinel files) was designed and implemented but never tested end-to-end with a real Claude Code session. The old pane-output polling was confirmed broken (TUI alternate screen buffer). The new approach should work but hasn't been validated.

**Risks:**
- SessionStart hook might fire before or during workspace trust dialog (not after)
- VCO_AGENT_ID env var might not be inherited by hook subprocesses
- Signal file path (`/tmp/vco-agent-{id}.state`) could have permission issues
- The 3-second sleep + Enter for trust dialog is still a race condition

**Test plan:** Launch a single Claude Code session manually in a scratch dir with the hooks configured, verify the sentinel file gets written on startup and after each response.

### Task Queue Drain Timing

**Problem:** The idle watcher (`_watch_idle_signals`) polls the sentinel file every 1 second. When it sees "idle", it drains the queue and clears the file. But:
1. If two tasks are queued rapidly, only one gets sent (queue drains one at a time, which is correct)
2. The agent might still be processing when the watcher reads a stale "idle" from a previous cycle
3. Clearing the signal file after drain means we might miss the next idle signal if it fires between clear and next poll

**Mitigation:** The drain sets `_is_idle = False` before sending, and clears the file. The next Stop hook writes "idle" again after the agent finishes processing. This should work but has a theoretical race window of ~1 second.

### Discord Channel Creation for Task Agents

**Problem:** `setup_task_channel()` creates channels under a `vco-tasks` category. But `discord_send.py` (used by `/vco:send`) looks for channels named `task-{id}` or `agent-{id}` — it doesn't filter by category. If there are multiple channels with the same name across categories, it might post to the wrong one.

**Fix:** `discord_send.py` should filter by `vco-tasks` category, same as `ask_discord.py` filters by project category.

### Research Agent Scripts Not Validated

**Problem:** The research scripts (`validate_report.py`, `verify_citations.py`, `source_evaluator.py`, `citation_manager.py`) were downloaded from 199-biotechnologies/claude-deep-research-skill. They parse Python AST fine but:
1. They may have internal dependencies (import each other) that break when deployed to a flat `scripts/` directory
2. They may expect specific file paths or directory structures
3. They haven't been tested with an actual REPORT.md output

**Test plan:** Run `python3 scripts/validate_report.py --report REPORT.md` with a sample report to verify it works standalone.

### Pre-existing Test Flake

**Problem:** `test_company_agent.py::TestCrashRecovery::test_crash_recovery_restores_count` fails intermittently. The background `_drain_events` task races with manual `process_next_event()` calls in the test. One event gets consumed by the drain before the test can process it.

**Root cause:** Test calls `start()` which spawns `_drain_events` background task, then immediately posts events and tries to process them manually. The drain task steals one.

**Fix:** Test should either not start the drain task, or use only drain-based processing (not manual).

### `_build_launch_command` Still Has Trust Dialog Workaround

**Problem:** `_launch_tmux_session()` still has `await asyncio.sleep(3)` + `pane.send_keys("", enter=True)` to auto-accept the workspace trust dialog. This is a race condition — if Claude hasn't loaded in 3 seconds, the Enter goes nowhere. If it loaded faster, the Enter might hit something else.

**Proper fix options:**
1. Pre-trust the clone/scratch directory in Claude Code settings
2. Find a Claude Code flag that skips the trust dialog
3. The signal-based approach (wait for SessionStart) should handle this — if SessionStart fires after trust, we know it's accepted. But the blind Enter before that is still a problem.

### `[CMD:...]` Parsing Code Is Dead Weight

**Problem:** `StrategistCog._execute_actions()` parses `[CMD:hire/give-task/dismiss]` tags from Strategist responses. This was built before realizing the Strategist should use `vco` CLI via Bash instead. The parsing code works but is the wrong approach.

**Action:** Remove `_execute_actions()` and the `_CMD_PATTERN` regex from `strategist.py` when v3.0 CLI commands are ready. Keep the agent management section in the persona but rewrite it to reference `vco` CLI commands instead of `[CMD:...]` tags.

### CompanyAgent Callback Wiring Is Over-Engineered

**Problem:** `CompanyAgent` has `_on_hire`, `_on_give_task`, `_on_dismiss` callbacks wired in `bot/client.py on_ready`. These exist because the Strategist can't call CompanyRoot directly. With v3.0 (CLI-first), all of this wiring disappears — the Strategist just runs `vco hire` via Bash.

**Action:** Keep for now (it works if triggered via events). Remove when v3.0 CLI commands replace the event-based flow.
